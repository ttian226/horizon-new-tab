# v0.1.4 — 留存周期

> 开发周期目标：把扩展从"功能齐全的工具"推向"用户日开仪表盘"。三个核心改进 + 一个基建。
> 周期开始：2026-05-26
> 目标发版：TBD（参考 v0.1.3 的 3 周节奏）

---

## 本期目标（Why）

v0.1.2 — v0.1.3 的迭代解决了"产品看着没动"的感知问题（What's New 弹窗、壁纸轮换）。
v0.1.4 解决"老用户为什么留下来"的实际问题：

- 让 todos 成为日常打开的理由（功能存在，但被埋）
- 修掉 OAuth token 静默失效导致的"我以为还在同步、其实早断了"
- 给"每天打开"这个习惯一个最小化的反馈机制（streak）

直接服务 12 人样本里"5 人最近 3 个月没登录"的留存空洞。

### 🎯 P0 — 必做（核心留存）

- [ ] **Todos 上首页主视觉**
  - 现状：todos 藏在 widget / 侧栏，新用户/低频用户不知道
  - 目标：未完成 todos 在新标签页一眼可见（"你今天还有 N 件事"）
  - 待定：是直接展示一列、还是悬浮在时钟下方、还是用 widget 顶部摆位
  - 影响范围：`App.tsx` 主视图 + 可能调整 `TodoWidget` / 新增组件
  - 验收：登录用户开新标签能立刻看到未完成项，无需点击/滚动

- [ ] **OAuth token 静默续期**
  - 现状：怀疑用户掉同步后没察觉（12 人中 5 人最后登录 2026-02 之前）
  - 排查：检查 `src/services/auth.ts` 当 Firebase auth state 变 null 但用户没主动登出时的处理
  - 实现：token 失效时静默调 `chrome.identity.getAuthToken({ interactive: false })` 续期；失败再降级到提示用户重登
  - 验收：故意把 dev Firebase 的 session 弄过期，重新打开扩展应该无感恢复同步

## 风险 & 开放问题

1. **Todos 上首页的 UI 设计**：直接展示 vs 入口卡片 vs 浮动卡片？需要先做几个草图/截图比较
2. **OAuth 续期实现细节**：Chrome `identity.getAuthToken({ interactive: false })` 在 token 完全过期时可能仍要弹窗。需要实测各类失效场景
3. **Streak 是 localStorage 还是 Firestore**：本地存简单但跨设备就归零；同步到 Firestore 更准但多一次写入。倾向先 localStorage，体验跑通再考虑同步
4. **dev Firebase 的壁纸数据**：之前讨论用「dev 直读 prod 公开 collection」最省事，但 todos / streak 这类用户私有数据 100% 隔离。需要在代码里确认 wallpapers 读取路径不被 env 切换影响（如果它们用同一个 Firebase app，会自动跟着环境走，那壁纸 dev 就要自己 seed 了）
