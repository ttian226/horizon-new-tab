# v0.1.5 — Task 便签画布（Task-backed Sticky Notes）

> 周期开始：2026-05-29
> 主题：**把 Notion task 被忽略的"页面正文"放大成贴在新标签上的便签**
> 分支：`feature/v0.1.5`（起点 main `b7bf61e`）

---

## 战略定位（为什么是这个主题）

延续 v0.1.4 的"Horizon Tab = Notion 工作流的浏览器新标签前哨"。

**洞察**：Notion task 的「页面正文」（page body）几乎没人用——你自己、SyncTask iOS 都忽略它。与其再单独做一个鸡肋的 Markdown Notes（重复造轮子、违背主线），不如**把这个已存在却被埋没的正文放大**：

**「一张便签 = 一条 Notion task 的正文。在新标签上编辑便签 = 编辑那条 task 的正文。」**

- 零新数据模型：便签内容就是 task 的 page body，复用 v0.1.4 的 token / hook / schema
- 发挥新标签页独有优势：一块"抬眼就看见"的环境屏幕，便签贴在这上面才有意义（Notion 做不到）
- 两条腿互补：**列表（todo 面板）管结构化任务，便签管"想一直盯着"的少数 task**
- 顺带高分享性：磨砂半透明便签贴在壁纸上，适合截图（小红书线）

---

## 核心模型（已定）

| 维度 | 决定 | 理由 |
|---|---|---|
| **便签内容** | = Notion task 的 page body，**纯文本**（不做 Markdown）| MD↔Notion block 格式不兼容；纯文本够用且砍掉转换器 |
| **读** | `GET /v1/blocks/{pageId}/children` → 抽文本 → textarea | 简单 |
| **写** | textarea 内容按换行切段 → 生成 `paragraph` block → `PATCH /v1/blocks/{pageId}/children` | 无"整体替换正文"接口，只能重写段落 |
| **富内容护栏** | 正文空/纯段落 → 可编辑；含非段落块（图片/列表/表格/toggle）→ **只读 + "在 Notion 编辑"** | 重写会误删富内容，必须保护 |
| **便签来源（选择模型 A）** | 列表里每条 task 一个"钉到桌面"动作 → 变便签；取消钉收回。新建便签 = 新建 task | 列表当大本营，改动最小、心智清晰；保持"便签必有对应 task"不变式 |
| **便签布局（位置/大小/颜色）** | **本地存**（按设备，chrome.storage / localStorage，复用 GlassCard 套路）| Notion 存不了布局；像素坐标跨屏/跨设备不通用 |
| **正文拉取时机** | 懒加载（钉出/打开便签时才拉，不为所有 task 预拉）| 省 Notion rate limit（3 req/s）|
| **显示位置** | **便签只在 work mode 显示**（复用现有可拖拽 widget 画布）| 避免常驻新标签糊屏；work mode 本就是"摆 widget"的场景 |
| **旧 Markdown Notes** | **彻底移除**（不保留本地 MD 编辑器）| 鸡肋且违背主线；便签 = task 正文取而代之 |

### M2 — 便签写回（编辑）

- [ ] **纯文本编辑**：textarea 编辑 → 切段 → 写回 Notion 正文（乐观更新 + 失败回滚，沿用 v0.1.4 模式）

- [ ] **新建便签 = 新建 task**：在画布新建便签 → `POST /pages` 建 task（标题 + 正文 body）

## 不做 / 待定（推后或需决策）

| 项 | 说明 |
|---|---|
| **Notion OAuth 一键授权** | **最大增长杠杆**（手动贴 token + DB ID 门槛高）。工程量大（要 Cloud Function 做 code 交换）。是否抢在便签前/并行做，**待你决策** |

| **完整 block 类型读渲染** | M1 先纯文本；富 block 的只读渲染（标题/列表/代码）可后做 |

## 风险 & 开放问题

1. **Notion rate limit (3 req/s)**：每张便签一次正文 fetch，多便签 + 编辑写回可能撞墙 → 懒加载 + 请求节流。
2. **写回误删富内容**：核心风险，靠"富内容护栏"兜底；要把"只读"的判定做扎实（检测 blocks 是否全为 paragraph/空）。
3. **便签与现有 work-mode widget 共存**：已定便签只在 work mode、复用同一拖拽画布；要确保和 todo widget 同屏不冲突（层级、默认位置、bring-to-front）。
4. **跨设备/跨分辨率布局**：本地像素坐标在大屏/小屏错位 → 先按设备本地存，接受不同设备各自布局。
5. **糊屏**：多张彩色便签破坏干净视觉 → 显隐开关 + 数量上限 + 半透明克制。
