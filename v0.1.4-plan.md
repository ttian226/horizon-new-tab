## 战略定位（为什么是这个主题）

新标签页扩展做 todo 的太多了（Momentum / Tabliss / Bonjourr），重做一个无意义。Horizon Tab 的差异化路径：

**「如果你是 Notion 用户，Horizon Tab 就是你的 Notion 在浏览器里的前哨。」**

- 复用用户在 Notion 已有投资，不再要求把数据搬到我们的系统
- 我们只做"展示 + 操作转发"，task 增删改的真源是 Notion
- 维护面骤减（不用做提醒/优先级/视图，Notion 已经做好了）

长期方向（v0.1.5+）：Notion 页面当 Notes 内容源 / Notion DB 当可摆 widget。本期只做 Tasks 一项。

## 关键设计决策（已定）

| 决策 | 选择 | 理由 |
|---|---|---|
| **Lists 处理** | 抛弃 lists 概念 | Notion DB 无 list 抽象；现有 lists 是被埋的低使用率功能 |
| **两种模式共存** | 未连 Notion → 本地 todos；连了 Notion → Notion 接管 | 不强迫所有用户用 Notion；保留扩展对纯本地用户的可用性 |
| **历史本地 todos** | 启用 Notion 时**保留**在本地，但不显示 / 不同步 | 防止删数据焦虑；用户停用 Notion 时可恢复 |
| **Token 存放** | `chrome.storage.local` | 敏感，不进 Firestore |
| **Notion 字段映射** | 严格按 SyncTasks 现有 schema（`Name` / `Status` / `Due date` / icon）| 不污染你现有 iOS Widget 工作流 |
| **同步触发** | 开新标签 fetch + 5min poll + window focus refresh；写操作直接 push | M1/M2 实测：直 fetch/push 够用，未上 debounce |
| **行点击语义** | 本地：toggle done；Notion：toggle done + 推送（M2 起）| 最高频操作给主点击；开 Notion 页降级到 hover ↗ |
| **Notion 视图筛选** | 仅 Status + Due date 两轴的开关 + Sort，砍组合 builder | API 不暴露 View filter；这两轴已是 DB 全部可筛维度 |

## 风险 & 开放问题

1. **Notion API rate limit (3 req/sec)**：开新标签连续拉 + 多次 push 可能撞墙。需要 debounce + 请求队列。
2. **chrome.storage.local 的 token 安全性**：本地存敏感数据有风险（恶意扩展可能扫）。但短期内自用，可接受。多用户 OAuth 版本上线后会用 server-side flow 取代。
3. **冲突的 last-write-wins 漏数据**：极端情况（同一秒两端都改）会丢一端的改动。个人用够了，但写文档时要说明。
4. **现有 prod 用户升级体验**：v0.1.3 用户升级后会看到 What's New 弹窗讲 Notion，但未启用 Notion 时所有行为不变（本地 todos 继续用）。要确保 What's New 文案别让人觉得"我必须接 Notion 才能用"。
5. **dev Firebase 的壁纸数据空缺**：v0.1.3 留下的问题，本期顺手解决 —— 让 wallpapers fetch 始终走 prod Firestore，因为是公开只读数据。
